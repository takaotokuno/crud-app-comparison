### Next.jsフロントエンドの初期化
ルートリポジトリで実行

```bash
npx create-next-app@latest frontend
```
※devcontainerにおける、node.jsのバージョン関係で下記エラーが発生したため、バージョンを20→22に変更
```
Error [ERR_UNKNOWN_BUILTIN_MODULE]: No such built-in module: node:sqlite
```
外部パッケージなしでSQLiteが使えるようにnode.jsにはver22.5.0からnode:sqliteモジュールが追加された
最新のcreate-next-appを取得し実行したため、それが内部で使う依存パッケージが開発環境に不足しているとエラーとなる。

```bash
npm run dev
```
で起動できる。ポート番号はデフォルトで3000

### .NETバックエンドの初期化

ソリューションファイルを作成
```bash
dotnet new sln -n AspNetNextApp
```

プロジェクトを作成
```bash
dotnet new webapi -n AspNetNextApp.Api
```

ソリューションにプロジェクトを追加
```bash
dotnet sln add AspNetNextApp.Api/AspNetNextApp.Api.csproj
```

起動確認
```bash
dotnet run --project AspNetNextApp.Api
```

ポートはデフォルトでhttp:5000, https:5001
Docker実行時は、コンテナは8080で待ち受けホスト側で5000で公開

## Next.jsと.NETはどう動いているか？

npxコマンドを実行すると、npm registry `http://registry.npmjs.org` に、当該パッケージを探しに行く
運営主体はGitHub、npmをGitHubが買収したのだとか。ちなみにGitHubの親会社はMicrosoft

### Next.jsの初期ディレクトリ構成
```
frontend/
    app/
        page.tsx
        layout.tsx
        globals.css
    public/
    package.json
    package-lock.json
    next.config.ts
    tsconfig.json
    eslint.config.mjs
```

app/ 画面やルーティングを置く場所
app/page.tsx トップページ
app/layout.tsx 全ページ共通レイアウト
public/ 静的ファイル置き場
package.json npmスクリプトと依存パッケージ定義
next.config.ts Next.jsの設定
tsconfig.json TypeScript設定

```bash
npm run dev
```
これは`package.json`の`scripts.dev`を実行している。開発用コマンド。本番ではstart。

`next`パッケージが以下の一連の流れを統括する。
最終的な成果物は、`.next/`に格納される。
```
Next.jsのディレクトリ構成を読む
↓
ルーティング構造を解釈する
↓
Reactコンポーネントをサーバー用・ブラウザ用に分ける
↓
TypeScript/JSXなどを変換する
↓
Node.jsで実行できるサーバー成果物と、ブラウザで実行できるJS/CSSを作る
```


Dockerfileの構成は3段階
```
deps    : package-lock.jsonに基づいてnode_modulesを作成
    node:22-bookworm + package.json + node_modules
    ※

build   : Next.js アプリをビルドする
    node:22-bookworm + node_modules + ソースコード + .next

runtime : ビルド済みアプリを本番起動する
    node:22-bookworm + 本番起動に必要なものだけ
```
ただ、この程度の規模ならChatGPT曰く、別にdepsとbuildは分けなくても成立するらしい。
重要なのは、`npm ci`の後に`COPY`すること。キャッシュ効率が悪くなるため

OK例
```
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
```

NG例
```
COPY . .
RUN npm ci
RUN npm run build
```

### .NET側の動き

.NETでは、ソリューションファイルとプロジェクトファイルの2種類が出てくる
プロジェクトファイルは.NETのバージョン、依存NuGetパッケージを管理する

```bash
dotnet run --project AspNetNextApp.Api
```
によって`.csproj`のアプリをビルドして起動できる。※開発用コマンド

.NETではnpmと違い、プロジェクト直下に`node_modules`のような依存フォルダはできない。
NuGetパッケージはグローバルキャッシュの保存される。
Linuxだと概ね

```
/home/vscode/.nuget/packages/
```

nugetの取得元は
```
https://api.nuget.org/v3/index.json
```
こっちは当然Microsoft運営

### Node.js/npmと.NETの共通点・違い

どちらも、定義ファイルを読み、依存パッケージを復元し、ソースコードをビルド・実行、という流れは同じ。

| Node.js | .NET |
| ---- | ---- |
| package.json | .csproj |
| package-lock.json | packages.lock.json/project.assets.json |
| node_modules | ~/.nuget/packages/ |
| npm install/ npm ci | dotnet restore |
| npm run dev | dotnet run |
| npm run build | dotnet build / dotnet publish |

Node.jsの考え方
JavaScriptの世界では、

- 小さなパッケージを大量に組み合わせる文化
- フロントエンドのビルドツールが頻繁に変化
- グローバルに入れるとプロジェクト間で衝突しやすい

ので自己完結型になった。Webフロントエンドの変化の速さへの対応結果


.NETの考え方
.slnと.csprojが分かれている
パッケージがグローバルキャッシュ
ビルドはMSBuildが管理
.NETはMicrosoftのアプリケーション開発基盤として、企業向け大規模構成が多かった
複数のプロジェクトが共通のパッケージを使うことが多かった

package.json: パッケージ定義+スクリプト
.csproj: プロジェクト定義+依存関係+コンパイル設定

余談　.NET Core以降ではSDK-style projectという形式が標準で、
プロジェクトは以下の.csファイルが自動的にビルド対象になった
Visual Studio前提のプロジェクト形式、Windows前提、CLIだけで完結しにくかった
